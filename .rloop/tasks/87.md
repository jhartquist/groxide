Replace the fragile blank-line-counting logic in annotate_reexport with a structured approach.

## Problem

`annotate_reexport()` (in reexport.rs after task 77) reconstructs rendered output by counting blank lines to find an insertion point:
```rust
if blank_count == 2 { insert_pos = Some(i + 1); break; }
```
This is fragile — if text rendering format changes, it breaks silently.

## Fix

Have the text renderer accept re-export annotation as context, so it renders the annotation inline instead of post-hoc string surgery.

1. Add an optional parameter to `render_text()`:
   ```rust
   pub(crate) fn render_text(
       display: &DisplayItem<'_>,
       limits: &DisplayLimits,
       reexport_info: Option<(&str, &str)>,  // (original_path, source_path)
   ) -> String
   ```
2. When `reexport_info` is Some, the renderer inserts the re-export annotation line (e.g., `"re-exported as crate::path  (from source_crate)"`) at the correct position during rendering — it knows the structure.
3. Remove `annotate_reexport()` entirely
4. Update the call site in render/dispatch.rs to pass the info through

## Important

- The exact output format for re-export annotation should match what `annotate_reexport()` currently produces
- Snapshot tests will catch any differences
- If the parameter feels awkward, consider a `RenderOptions` struct instead of adding parameters
