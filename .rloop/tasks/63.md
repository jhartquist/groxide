## Automatically resolve re-exports when path lookup fails

**Plan reference:** docs/plans/2026-03-05-cli-redesign-implementation.md — Phase 5, Task 16

When `grox egui::ViewportInPixels` fails because the item is re-exported from epaint, groxide should follow the re-export chain and resolve it transparently.

### Background

The existing code in `src/lib.rs:564-609` (`try_follow_reexport`) handles re-exports for items that ARE found but are stubs. This task extends that to handle items that are NOT found — the path exists conceptually as a re-export but the index doesn't contain it at that path.

### What to do

1. **Understand the problem:** When crate A re-exports an item from crate B (`pub use B::Item`), rustdoc JSON for crate A may include the re-export as a stub with `reexport_source` pointing to B. But the query engine may not find it because:
   - The path in the index might be the original (B::Item) not the re-exported (A::Item)
   - Or the re-export stub exists but with limited info

2. **Approach: search the index for matching re-export stubs on NotFound**

   In `src/lib.rs`, after `resolve_item` returns `NotFound`:
   - Extract the last segment of the query (the item name)
   - Search the current crate's index for items with `reexport_source` set
   - If any re-export stub matches the query path, follow it using the existing `try_follow_reexport` mechanism
   - If the item name exists in a dependency index, suggest: "hint: did you mean {dep}::{Item}?"

3. **Simpler first approach:** Before going cross-crate, check if the item name exists in the current crate's index under a different path. If found, suggest:
   ```
   not found: egui::ViewportInPixels
   hint: did you mean epaint::ViewportInPixels? (re-exported by egui)
   ```

4. **Test with fixture crate:** The fixture has `reexports` module that re-exports from `inner`:
   - `groxide_test_api::reexports::Helper` re-exports `inner::Helper`
   - Test that querying the re-exported path resolves correctly

5. **Write integration test:**
   ```rust
   #[test]
   fn reexport_resolves_transparently() {
       let output = grox()
           .arg("groxide_test_api::reexports::Helper")
           .output()
           .expect("command runs");
       assert!(output.status.success());
       let stdout = String::from_utf8_lossy(&output.stdout);
       assert!(stdout.contains("Helper"));
   }
   ```

6. **Cross-crate re-export resolution** (stretch goal): When the item truly lives in a different crate (like epaint), load that crate's index and resolve there. This requires the context (ProjectContext) to find the dependency. Mark this as a follow-up if too complex for this task.
